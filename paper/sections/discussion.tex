\section{Discussion}
\label{sec:discussion}

\paragraph{Implications for practice.}
Read together, RQ1 and RQ3 point to the same conclusion: abstraction cost is not
a fixed, uniform overhead but a cost specific to a particular \emph{access pattern},
and it concentrates in the pattern that assembles a nested object graph. The
deep/nested fetch requires the access layer to hydrate rows into objects and resolve
associations (the object-relational impedance mismatch~\cite{ireland2009impedance}
made concrete), not merely to issue SQL, and this is exactly where the
data-mapper ORMs are most expensive: Sequelize, TypeORM, Objection, and
especially MikroORM (541~req/s) trail the native \texttt{pg} driver
(3{,}399~req/s) by up to $6.3\times$ on PostgreSQL, the widest spread in the
study (Table~\ref{tab:deep_fetch}), with a comparable $4.3\times$ spread on
MySQL. The tail shows the same pattern: MikroORM's deep-fetch p99 is 111~ms
against 21~ms for \texttt{pg} on PostgreSQL (Table~\ref{tab:deep_fetch}).
A team whose hot path assembles nested object graphs therefore pays the
largest abstraction cost precisely on its highest-value traffic. Point reads
and aggregation are the opposite case: dominated by connection and parsing
cost that every layer shares, point reads spread only $3.5\times$/$2.6\times$;
aggregation spreads just $1.8\times$/$1.4\times$ once every adapter, native
drivers included, is forced onto the same raw correlated-subquery plan; even then
a roughly twofold gap separates the fastest and slowest layers, far less than on the
deep fetch but not negligible. Prisma shows that an ORM
need not be slow, consistent with evidence that ORM cost is highly sensitive to
implementation and tuning~\cite{vanzyl2009optimisations}: its compiled query engine
is statistically tied with the native driver
on the deep fetch (3{,}531 vs.\ 3{,}399~req/s) and matches or exceeds
native on aggregation on both engines (8{,}034 vs.\ 7{,}086~req/s on PostgreSQL).
That parity is not free, however: at the \emph{same} ten-connection pool, Prisma's
process draws roughly five times the application-side CPU of the other layers on the
deep fetch ($471\%$; Section~\ref{sec:results}), trading CPU for
latency~\cite{bonvoisin2024orm}, a cost a throughput-only benchmark would miss. The
parity is also implementation-relative: because our native adapters use idiomatic
parameterized queries rather than tuned prepared statements, the finding is best
stated as ``under this implementation and resource model, a query-engine ORM can
match or exceed the native-driver adapter, at substantially higher application-side
CPU,'' not as an intrinsic ceiling. Round-trip count alone does not set throughput:
Prisma issues four queries for the deep fetch (Table~\ref{tab:query_counts}) yet
matches the native driver, while Objection issues the same four and is among the
slowest, so how a layer pipelines and materializes its queries matters as much as how
many it sends. Knex and Drizzle sit in a consistent middle ground on every pattern (never fastest, never
among the slowest data-mapper ORMs), which is what a query builder and a
non-data-mapper, lightweight ORM should deliver.

\paragraph{Writes: an engine-bound cost.}
RQ2 has a clear answer on inserts, within the relaxed-durability configuration used
here. MySQL write throughput is low and flat across all eight layers
(\char`~745--939~req/s, a $1.3\times$ spread; Table~\ref{tab:write}), while the
identical comparison on PostgreSQL spreads $4.7\times$ (native 7{,}040
vs.\ MikroORM 1{,}494~req/s). The access
layer barely matters for MySQL inserts because it is not the bottleneck there:
InnoDB's per-insert write path (redo-log generation, the doublewrite buffer, and
clustered-index, secondary-index, and undo maintenance), work internal to the engine
rather than to any access layer~\cite{harizopoulos2008oltp}, governs end-to-end
latency regardless of how a query reaches it. This holds even though the redo log is
flushed lazily under our relaxed-durability configuration
(Section~\ref{sec:methodology}): the remaining per-row engine work still dominates. This finding also warrants
caution: a write-throughput comparison run on a single engine invites the wrong
generalization in either direction (``the access layer is irrelevant to writes''
from MySQL alone, or ``the access layer matters a great deal for writes'' from
PostgreSQL alone), and an engine-only benchmark such as~\cite{salunke2024pgmysql}
cannot expose this interaction because it omits the access layer entirely. Only the
dual-engine design used here makes the engine-write-path explanation visible
instead of confounded with layer choice; that the underlying database technology
modulates the cost of a given mapping strategy is consistent with earlier
quantitative ORM work~\cite{lorenz2017orm}. The finding is robust to the durability
configuration (Table~\ref{tab:durability}): re-measured under full default durability
and under symmetric relaxation (with MySQL's doublewrite buffer also disabled), MySQL
inserts stay flat (a $1.1$--$1.3\times$ layer spread) while PostgreSQL stays
layer-dependent ($4.0$--$4.8\times$), so the asymmetric relaxation of the primary run
does not manufacture the result. Database-side CPU corroborates it: during MySQL
inserts the \texttt{mysqld} process is CPU-bound ($\approx115$--$133\%$) and roughly
equal across layers, whereas on PostgreSQL database CPU tracks throughput, confirming
the bottleneck is engine-internal on MySQL.

\paragraph{Methodological lessons: a checklist for access-layer benchmarking.}
Reaching the figures above meant confronting four confounds that would otherwise
have silently corrupted the comparison. Their provenance differs. One, the
aggregation fan-out, is a \emph{correctness}
bug that our automated cross-check caught (the same check, which compares each
adapter's output against a native-driver reference on four of the five endpoints,
also caught a driver result-shape mismatch). The other three return \emph{correct}
results and were therefore invisible to any equality check; they were found only by
performance analysis. We regard the resulting checklist as reusable:
\begin{itemize}[leftmargin=*,noitemsep,topsep=2pt]
  \item \textbf{Aggregation fan-out.} A naive join between \texttt{posts} and
        \texttt{comments} evaluated before the \texttt{SUM} multiplies each
        post's view count by its number of matching comment rows, inflating
        the aggregate, a correctness bug invisible unless every layer's
        output is checked against a reference, not a performance one.
  \item \textbf{OFFSET vs.\ keyset pagination.} An early, OFFSET-based design
        for the range-scan endpoint collapsed specifically on MySQL at
        realistic offsets, which would have been misread as an access-layer
        weakness rather than what it is: a pagination-strategy cost that
        InnoDB pays more dearly for than PostgreSQL. Rewriting every adapter
        around keyset pagination (\texttt{WHERE id < cursor ORDER BY id DESC
        LIMIT n}) removed the confound.
  \item \textbf{Write-workload contamination.} The insert endpoint grows
        \texttt{posts} on every call; because engines and layers commit at
        different rates, later cells in a run would scan a differently sized
        table depending on run order rather than on layer identity, unless
        benchmark-inserted rows are deleted before every cell.
  \item \textbf{Query-formulation confounds.} A first attempt at the
        aggregation endpoint pre-aggregated \emph{all} of \texttt{comments}
        before filtering to one author, so every request re-scanned the whole
        table regardless of which author was requested, a query-plan
        artifact masquerading as an access-layer effect~\cite{leis2015optimizers},
        fixed by rewriting
        the query as a correlated subquery touching only the target author's
        rows.
\end{itemize}
None of these four is specific to our schema; each is a generic failure mode
of the genre, extending the general database-benchmarking pitfalls cataloged
by Fruth et al.~\cite{fruth2022tail} and Raasveldt et al.~\cite{raasveldt2018fair}
to the access-layer-comparison sub-genre specifically. We regard the checklist as a contribution in its own right,
reusable by anyone comparing drivers, query builders, or ORMs: each confound
is easy to introduce by accident and hard to detect (the correctness ones without
an automated cross-check, the performance ones without profiling every cell), and at
least one such confound plausibly explains part of the gap between some
vendor-reported and independently verified figures in this space.

\paragraph{Practical guidance.}
The practical synthesis follows directly from RQ1--RQ3. Where a service's hot
path is dominated by deep/nested fetches, the access layer is consequential
and a thin layer (the native driver, a query builder, or a query-engine-based
ORM such as Prisma) is the safer default, since a heavyweight data-mapper ORM
concentrates its largest penalty exactly on that traffic. Where the workload
mix is read-simple (point reads, keyset scans) or write-bound, especially on
MySQL where the engine floor dominates layer differences regardless of choice,
the productivity, type safety, and migration tooling of a full ORM cost little
in relative throughput and are reasonable grounds for the decision on their
own. A pragmatic middle path, already forced onto every layer by our
aggregation endpoint, is to keep the ORM for the bulk of the application and
fall back to raw SQL or the native driver only for the specific endpoints that
assemble large object graphs; orthogonally, second-level caching can absorb part of
an ORM's read cost where the workload tolerates staleness~\cite{chen2016cacheoptimizer}.

These recommendations rest on one schema, one hardware configuration, a single-host
local-database deployment with relaxed durability, and the library versions pinned in
the replication package. A production deployment adds network latency, TLS, a reverse
proxy, a separate database host, connection poolers, read replicas, default
durability, and larger or more varied payloads, any of which can shift the balance;
Section~\ref{sec:threats} sets out how far these results can be expected to generalize.
