\section{Conclusion and Future Work}
\label{sec:conclusion}

We set out to replace vendor-benchmark folklore with a neutral, reproducible
comparison of relational database access layers in Express.js. Our harness serves
an identical application through the full taxonomy---native drivers, a query
builder, and six ORMs---across PostgreSQL and MySQL, over five access patterns,
and reports throughput and tail latency together. This closes the four gaps that
characterize prior art: the absence of MySQL from access-layer comparisons, the
lack of a neutral full-taxonomy study, the separation of throughput from tail
latency, and the omission of the Express HTTP layer.

The empirical answer is that the cost of abstraction is concentrated, not uniform.
It is negligible for point reads and for aggregation issued through a raw-SQL path,
but it grows sharply---up to $6.4\times$ between the native driver and the heaviest
data-mapper ORM---for the deep/nested fetch, where the layer must assemble a nested
object graph on the application side. The relative ordering of layers is stable
across the two engines, and a competitively fast modern ORM (Prisma) shows that
``ORM'' need not imply large overhead. Writes are the exception: MySQL's commit
path makes insert throughput nearly layer-independent, a caution against
generalizing single-engine write results. Beyond these findings, the correctness
cross-check that underpins the harness surfaced a reusable checklist of
benchmarking pitfalls---aggregation fan-out, \texttt{OFFSET}-versus-keyset
pagination, write-workload contamination, and query-formulation confounds---that
silently distort naive access-layer benchmarks.

Several extensions would strengthen the evidence. A concurrency sweep would locate
the saturation point of each layer and separate latency-bound from throughput-bound
regimes; per-cell CPU and memory sampling would add a resource dimension; and a
fixed-rate, two-machine cross-run would bound coordinated-omission and
observer-interference effects. On the analysis side, we plan pairwise significance
testing with effect sizes, and on the coverage side, additional engines and a
dedicated study of the N+1 penalty (naive versus eager loading per ORM) and of
cold-start latency. The complete replication package---harness, seed, adapters,
and analysis---is released under open licenses so that these extensions, and
independent re-runs on current library versions, are straightforward.
