\section{Threats to Validity}
\label{sec:threats}

\paragraph{Construct validity.}
Our dependent variables are HTTP-level throughput (req/s) and tail latency
(p99) measured at the Express endpoint, not query-execution time at the
driver or database boundary. This is deliberate---it captures the full request
round trip a deployed service incurs, including Express routing and JSON
serialization---but it means the overhead we attribute to the \emph{access layer}
may partly reflect these surrounding costs instead. We control for this by holding
those costs constant across conditions: every adapter satisfies a documented
\emph{adapter contract} (Section~\ref{sec:methodology}) that guarantees an
identical result shape to Express, so the JSON payload---and its serialization
cost---is the same regardless of which layer produced it. Observed differences
between layers are therefore attributable to how each layer reaches that shape,
not to what is returned.

\paragraph{Internal validity.}
A naively configured ORM can trigger N+1 queries and appear artificially slow for
reasons unrelated to its inherent cost; every adapter is required to use its
layer's documented, idiomatic eager-loading strategy on the deep-fetch endpoint
(Section~\ref{sec:methodology}). An automated correctness cross-check asserts that
all nine adapters return identical, normalized results---compared as a projection of
key fields on four of the five access patterns---before any timing run is trusted;
this is not a formality---it caught two defects
during development: a fan-out bug that inflated the \texttt{SUM(views)} aggregate
for every join-based adapter, and a result-shape parsing defect in one driver
binding, both fixed before the measurements reported here. Explicit warm-up phases
per (layer, engine, endpoint) cell absorb JIT, pool-fill, and plan-cache effects
before timing begins. Because the insert endpoint mutates the dataset,
benchmark-inserted rows are deleted before every cell, so every read and
aggregation measurement starts from an identical seeded table. The aggregation
endpoint forces all adapters onto the same correlated-subquery plan rather than
each adapter's first-instinct formulation, so it measures access-layer overhead,
not incidental differences in SQL shape. A residual risk is coordinated omission in
the load generator, which can under-report tail latency at
saturation~\cite{fritz2021tail}; a fixed-rate, constant-arrival cross-run is
planned to bound it.

\paragraph{External validity.}
All measurements come from a single schema, a single dataset size, one hardware
configuration, and a connection pool fixed at ten, against specific engine and
library versions (PostgreSQL~18.4, MySQL~9.7.1; Section~\ref{sec:methodology});
results may not transfer to larger working sets, other pool sizes, or other engine
versions, and library performance ages quickly. We do not claim the absolute req/s
figures generalize beyond our hardware; only the \emph{relative} ordering within an
engine is intended to travel. We mitigate this by pinning every dependency
version, publishing the deterministic seed and an automated environment capture,
and releasing the full harness so the matrix can be re-run on other hardware or
against newer releases. A further limitation is that the load generator and server
under test share a host---common practice in this genre, and it removes network
variance as a confound, but it can understate network-bound effects. A
two-machine cross-run over a dedicated link is planned to bound this.

\paragraph{Conclusion validity.}
We report medians of ten measured runs per cell and use the coefficient of
variation (CV) across runs as a stability signal (Table~\ref{tab:cv}). To guard
against reading noise as a ranking, we test each adjacently ranked pair on the
deep/nested fetch with a two-sided Mann--Whitney~$U$ test and report Cliff's~$\delta$
as the effect size (Table~\ref{tab:significance}); the coarse ordering is supported
with large effect sizes. Two bounds reinforce this: the widest spreads, such as
$6.2\times$ on the deep/nested fetch (Table~\ref{tab:deep_fetch_rps}), are an order
of magnitude larger than the run-to-run CV, and the ranking is preserved across the
full 1--256 connection sweep (Figure~\ref{fig:scaling}). Claims about very closely
ranked adjacent layers, where the effect size is small, remain appropriately
cautious. One caveat on the design: the ten runs of a cell share a single server
process lifetime (the server is booted once per layer/engine cell), so they are
repeated measures rather than fully independent replicates; the tests are best read
as within-process stability checks. This is unlikely to affect the conclusions,
since the between-layer gaps are roughly an order of magnitude larger than the
within-cell CV, but a design that reboots the server per replicate would be
stronger.
