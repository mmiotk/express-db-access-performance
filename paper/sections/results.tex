\section{Results}
\label{sec:results}

The tables below are generated directly by the harness and report throughput
(requests per second, higher is better) and tail latency (p99, milliseconds, lower
is better) per access layer and engine, one table per access pattern. All numbers
come from the run described in Section~\ref{sec:methodology} (seed of 2{,}000
authors, 100{,}000 posts and 1{,}000{,}000 comments; 50 concurrent connections;
median of ten measured runs after a warm-up pass; connection pool fixed at ten).
Absolute throughput is hardware-specific; the comparison of interest is
\emph{relative} across layers within an engine.

\subsection{RQ1: overhead relative to the native baseline}
The native \texttt{pg} driver leads on every PostgreSQL pattern, but the cost of
abstraction is strongly \emph{pattern-dependent} (Tables~\ref{tab:point_read},
\ref{tab:deep_fetch}, \ref{tab:aggregation}). On the \textbf{deep/nested
fetch} (the pattern that assembles a post, its author, and all comments with their
authors), Prisma reaches 3{,}521~req/s against native's 3{,}616, a gap of only a
few percent, and Knex (2{,}454) and the lightweight ORM Drizzle (2{,}098) remain
competitive, whereas the data-mapper ORMs are markedly slower: TypeORM 1{,}259,
Sequelize 1{,}066, Objection 1{,}009, and MikroORM 579, i.e.\ $6.2\times$ below
native. \textbf{Point reads} show a smaller $3.2\times$ spread (native 6{,}902 down
to MikroORM 2{,}141). \textbf{Aggregation} overhead is much reduced but not eliminated: because every
layer issues the same correlated-subquery plan through its raw-SQL path, the spread
narrows to $1.8\times$ native-relative (native 7{,}210 vs.\ Objection's 4{,}047),
or $2.0\times$ counting Prisma's 8{,}076, the highest of any layer. Even on this
identical query plan the layer still moves throughput about twofold, with Objection's
raw path the low outlier. Abstraction is thus inexpensive when the layer forwards a
query and costly when it must materialize a nested object graph.

\subsection{RQ2: engine dependence of the ranking}
The \emph{relative} ordering of access layers is largely stable across PostgreSQL
and MySQL: native drivers and Prisma lead, query builders and the lightweight ORM
form a middle band, and the data-mapper ORMs trail, on both engines (the
deep-fetch ordering, for instance, is preserved, with a $4.5\times$ spread on
MySQL versus $6.2\times$ on PostgreSQL). On MySQL, Prisma in fact edges past the
native \texttt{mysql2} driver on most patterns (e.g.\ deep fetch 3{,}514
vs.\ 2{,}324~req/s), so the native driver is not always the ceiling there. This
should be read cautiously: the native adapters issue parameterized \texttt{query()}
calls rather than reusable prepared statements, so they represent idiomatic rather
than maximally tuned native usage, and Prisma reaches this throughput at roughly
five times their application-side CPU (Section~\ref{sec:results}, resource
footprint). We still report all spreads native-relative for cross-pattern
comparability. The clearest
engine-specific effect is on
\textbf{inserts} (Table~\ref{tab:write}): on MySQL, throughput is uniformly low
across \emph{all} layers (roughly 750--930~req/s, a spread of only $1.2\times$), so
the InnoDB write path dominates and masks access-layer differences, whereas on
PostgreSQL the same pattern spreads $4.0\times$ (native 6{,}375 down to MikroORM
1{,}575). Thus for writes the engine, not the access layer, is the primary
determinant of throughput. The keyset range scan is healthy for eight of the nine
layers on both engines (Table~\ref{tab:range_scan}; native \texttt{mysql2}
2{,}901, \texttt{pg} 3{,}620, and a $\approx1.7\times$ spread among the remaining
layers), confirming that the earlier collapse we observed under \texttt{OFFSET}
pagination was a workload artifact rather than an engine or layer property
(Section~\ref{sec:discussion}). The one exception is MikroORM, whose range-scan
throughput drops to 676~req/s on PostgreSQL and 627 on MySQL (p99 of 263 and
323~ms), an anomaly we flag but do not fully isolate; it is consistent with the
per-request cost of materializing the returned page into MikroORM's identity map.

\subsection{RQ3: which patterns matter most}
Ranking the patterns by how much the layer choice moves throughput yields a clear
order. The deep/nested fetch is the most consequential ($6.2\times$ on PostgreSQL,
$4.5\times$ on MySQL). The keyset range scan has the next-largest \emph{nominal}
spread ($5.4\times$/$4.6\times$), but, unlike the deep fetch, where cost rises
gradually across the taxonomy, this is almost entirely a single-layer outlier:
excluding MikroORM, the range-scan spread is only $\approx1.7\times$. Inserts on
PostgreSQL follow ($4.0\times$; but only $1.2\times$ on MySQL, where the engine
dominates), then point reads ($3.2\times$/$2.4\times$). Aggregation is the least
consequential once every layer uses the raw-SQL path ($1.8\times$/%
$1.4\times$). In short, the more of the result graph the access layer must assemble
on the application side, the more the choice of layer costs, with MikroORM's
range-scan drop a layer-specific anomaly rather than a graded cost.

\subsection{Tail latency}
We report p99 for every pattern (the tail metric that governs perceived latency
under load); p50 and p90 are in the replication package. Tail latency tracks
throughput: the layers and patterns that lose throughput also exhibit the worst p99
(Tables~\ref{tab:deep_fetch}, \ref{tab:range_scan}).
The data-mapper ORMs' deep-fetch cells, in particular, combine the lowest
throughput with the highest p99, so the penalty a heavyweight ORM imposes on a
nested-fetch hot path is visible not only in mean capacity but in the tail under
load, the quantity that single-metric, throughput-only vendor benchmarks omit. These
p99 values come from a closed-loop, constant-concurrency generator and are not
equivalent to the tail a system exhibits under constant request arrival; an
open-loop, constant-arrival cross-run corrected for coordinated omission is a planned
extension (Section~\ref{sec:threats}), so we read the tail results as relative within
the closed-loop model.

\subsection{Measurement stability and significance}
The rankings above rest on stable measurements. Across the 40 PostgreSQL cells the
coefficient of variation of throughput over the ten runs is at most 10.7\%
(Table~\ref{tab:cv}), and across the 40 MySQL cells at most 7.1\%
(Table~\ref{tab:cv_mysql}); this global maximum is an order of magnitude below the
between-layer spreads (up to $6.2\times$). To confirm that neighboring ranks are not measurement
artifacts, we compare each adjacently ranked pair on the deep/nested fetch
(Table~\ref{tab:significance}): \emph{every} adjacent pair separates with a large
Cliff's~$\delta$ effect size ($\delta \geq 0.78$) and a Mann--Whitney~$U$
$p \leq 0.004$, including the two closest pairs (\texttt{pg}~$>$~Prisma,
3{,}616 vs.\ 3{,}521, and Sequelize~$>$~Objection, 1{,}066 vs.\ 1{,}009~req/s; both
$\delta = 0.78$, $p = 0.004$). The full ordering (native, Prisma, the query builder and lightweight
ORM, then the data-mapper ORMs) is therefore statistically supported at every step,
not merely in aggregate. Because the ten runs of a cell share one server-process
lifetime, these tests establish within-process stability rather than independent
replication (Section~\ref{sec:threats}).

\subsection{Scaling with concurrency}
Figure~\ref{fig:scaling} sweeps the deep/nested fetch from 1 to 256 connections. The
coarse three-band ordering (native and Prisma on top, the query builder and
lightweight ORM in the middle, the data-mapper ORMs below) holds at every operating
point at or above 8 connections; at a single connection the picture differs
(latency-bound, with several layers within noise of each other). Within the top
band, \texttt{pg} and Prisma alternate as the fastest layer: \texttt{pg} leads up
to 32 connections and Prisma from 64 onward, consistent with Prisma's higher parallelism
(and higher CPU use). Overall, the lightweight layers reach a markedly higher
throughput ceiling than the data-mapper ORMs across the load range, so the RQ1
finding is not an artifact of the single 50-connection operating point.

\subsection{Resource footprint}
Throughput is not the only cost (Table~\ref{tab:resources}). Prisma's near-native
throughput on the deep/nested fetch comes at a much higher CPU cost: measured at the
\emph{same} ten-connection pool as every other layer, its Node.js process draws
$469\%$ (about five cores) against the $\approx100\%$ (one core) of the others,
because Prisma runs a multi-threaded query engine (a Tokio runtime) in-process.
Because the pool is held constant, this is a property of the layer, not of a larger
connection pool. Prisma's throughput parity with native therefore trades CPU for
latency, which a throughput-only benchmark would not reveal. Two caveats apply: this
is \emph{application-process} CPU only (database-side CPU lives in the
\texttt{postgres}/\texttt{mysqld} process and is excluded for every layer alike), and
the multiplier is pattern-dependent (Prisma's aggregation cell draws only $\approx
200\%$). Memory differences are smaller. The native driver, Knex, and Objection are
lightest (204--213~MB peak RSS), while the remaining ORMs use more (251--278~MB).
Resource cost, like throughput, is therefore pattern- and layer-specific, and
matters when sizing horizontally scaled deployments, where per-instance resource
use governs how many replicas a given load requires~\cite{vaquero2011scaling}.
