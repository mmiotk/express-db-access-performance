\section{Results}
\label{sec:results}

The tables below are generated directly by the harness and report throughput
(requests per second, higher is better) and tail latency (p99, milliseconds, lower
is better) per access layer and engine, one table per access pattern. All numbers
come from the run described in Section~\ref{sec:methodology} (seed of 2{,}000
authors, 100{,}000 posts and 1{,}000{,}000 comments; 50 concurrent connections;
median of five independent replicates, each a freshly booted server process;
connection pool fixed at ten).
Absolute throughput is hardware-specific; the comparison of interest is
\emph{relative} across layers within an engine.

\subsection{RQ1: overhead relative to the native baseline}
The native \texttt{pg} driver leads or ties for the lead on the PostgreSQL patterns,
but the cost of abstraction is strongly \emph{pattern-dependent}
(Tables~\ref{tab:point_read}, \ref{tab:deep_fetch}, \ref{tab:aggregation}). On the
\textbf{deep/nested fetch} (the pattern that assembles a post, its author, and all
comments with their authors), Prisma reaches 3{,}531~req/s, statistically tied with
native's 3{,}399 (Section~\ref{sec:results}, significance), and Knex (2{,}424) and
the lightweight ORM Drizzle (2{,}061) remain competitive, whereas the data-mapper
ORMs are markedly slower: TypeORM 1{,}244, Sequelize 1{,}079, Objection 976, and
MikroORM 541, i.e.\ $6.3\times$ below native. \textbf{Point reads} show a smaller
$3.5\times$ spread (native 6{,}609 down to MikroORM 1{,}904). \textbf{Aggregation}
overhead is much reduced but not eliminated: because every layer issues the same
correlated-subquery plan through its raw-SQL path, the spread narrows to $1.8\times$
native-relative (native 7{,}086 vs.\ Objection's 3{,}895), or $2.1\times$ counting
Prisma's 8{,}034, the highest of any layer. Even on this identical query plan the
layer still moves throughput about twofold, with Objection's raw path the low outlier. Abstraction is thus inexpensive when the layer forwards a
query and costly when it must materialize a nested object graph.

\subsection{RQ2: engine dependence of the ranking}
The \emph{relative} ordering of access layers is largely stable across PostgreSQL
and MySQL: native drivers and Prisma lead, query builders and the lightweight ORM
form a middle band, and the data-mapper ORMs trail, on both engines (the
deep-fetch ordering, for instance, is preserved, with a $4.3\times$ spread on
MySQL versus $6.3\times$ on PostgreSQL). Prisma in fact matches or edges past the
native driver on the deep fetch on \emph{both} engines (on MySQL 3{,}495
vs.\ 2{,}287~req/s; on PostgreSQL it is statistically tied with \texttt{pg}), so the
native driver is not always the ceiling. This should be read cautiously: the native
adapters issue parameterized \texttt{query()} calls rather than reusable prepared
statements, so they represent idiomatic rather than maximally tuned native usage, and
Prisma reaches this throughput at roughly five times their application-side CPU
(Section~\ref{sec:results}, resource footprint). We still report all spreads
native-relative for cross-pattern comparability. The clearest engine-specific effect
is on \textbf{inserts} (Table~\ref{tab:write}): on MySQL, throughput is uniformly low
across \emph{all} layers (roughly 745--939~req/s, a spread of only $1.3\times$), so
the InnoDB write path dominates and masks access-layer differences, whereas on
PostgreSQL the same pattern spreads $4.7\times$ (native 7{,}040 down to MikroORM
1{,}494). Thus for writes the engine, not the access layer, is the primary
determinant of throughput. The keyset range scan is healthy for eight of the nine
layers on both engines (Table~\ref{tab:range_scan}; native \texttt{pg}
3{,}595, \texttt{mysql2} 2{,}965, and a $1.6$--$1.7\times$ spread among the remaining
layers), confirming that the earlier collapse we observed under \texttt{OFFSET}
pagination was a workload artifact rather than an engine or layer property
(Section~\ref{sec:discussion}). The one exception is MikroORM, whose range-scan
throughput drops to 640~req/s on PostgreSQL and 619 on MySQL (p99 of 94 and
92~ms), an anomaly we flag but do not fully isolate; it is consistent with the
per-request cost of materializing the returned page into MikroORM's identity map.

\subsection{RQ3: which patterns matter most}
Ranking the patterns by how much the layer choice moves throughput yields a clear
order. The deep/nested fetch is the most consequential ($6.3\times$ on PostgreSQL,
$4.3\times$ on MySQL). The keyset range scan has the next-largest \emph{nominal}
spread ($5.6\times$/$4.8\times$), but, unlike the deep fetch, where cost rises
gradually across the taxonomy, this is almost entirely a single-layer outlier:
excluding MikroORM, the range-scan spread is only $1.6$--$1.7\times$. Inserts on
PostgreSQL follow ($4.7\times$; but only $1.3\times$ on MySQL, where the engine
dominates), then point reads ($3.5\times$/$2.6\times$). Aggregation is the least
consequential once every layer uses the raw-SQL path ($1.8\times$/%
$1.4\times$). In short, the more of the result graph the access layer must assemble
on the application side, the more the choice of layer costs, with MikroORM's
range-scan drop a layer-specific anomaly rather than a graded cost.

\subsection{Tail latency}
We report p99 for every pattern (the tail metric that governs perceived latency
under load); p50 and p90 are in the replication package. Tail latency tracks
throughput: the layers and patterns that lose throughput also exhibit the worst p99
(Tables~\ref{tab:deep_fetch}, \ref{tab:range_scan}).
The data-mapper ORMs' deep-fetch cells, in particular, combine the lowest
throughput with the highest p99, so the penalty a heavyweight ORM imposes on a
nested-fetch hot path is visible not only in mean capacity but in the tail under
load, the quantity that single-metric, throughput-only vendor benchmarks omit. These
p99 values come from a closed-loop, constant-concurrency generator. To check that the
tail ranking is not an artifact of that model, we additionally drove the deep/nested
fetch on PostgreSQL under a constant offered request rate (open-loop, via autocannon's
\texttt{overallRate} over 256 connections; Table~\ref{tab:openloop}). The ranking
holds and sharpens: at every offered rate native \texttt{pg} has the lowest p99 and
the data-mapper ORMs the highest, and the ORMs saturate far earlier---MikroORM cannot
sustain 1{,}000~req/s and its p99 climbs to 3{,}150~ms while native \texttt{pg} stays
at 157~ms, a $\approx20\times$ tail gap. Under constant arrival the heavyweight-ORM
tail penalty is therefore \emph{larger} than the closed-loop figures suggest, not
smaller. This is a constant-arrival generator rather than a formally
coordinated-omission-corrected one (e.g.\ wrk2), which would report equal or higher
tails, so these open-loop numbers are conservative.

\subsection{Measurement stability and significance}
The rankings above rest on stable measurements. Across the 40 PostgreSQL cells the
coefficient of variation of throughput over the five independent replicates is at
most 6.9\% (Table~\ref{tab:cv}), and across the 40 MySQL cells at most 11.2\%
(Table~\ref{tab:cv_mysql}); this global maximum is well below the between-layer
spreads (up to $6.3\times$). To confirm that neighboring ranks are not measurement
artifacts, we compare each adjacently ranked pair on the deep/nested fetch with the
Mann--Whitney~$U$ test, Cliff's~$\delta$, and bootstrap 95\% confidence intervals
(Table~\ref{tab:significance}). \emph{Every} adjacent pair separates with a maximal
effect ($\delta = 1.00$) and $p = 0.012$, \emph{except} the top pair: native
\texttt{pg} and Prisma are statistically indistinguishable (3{,}399 vs.\ 3{,}531~req/s;
$\delta = 0.20$, $p = 0.68$, overlapping bootstrap CIs). Every other step in the
ordering---Prisma and \texttt{pg} at the top, then the query builder and lightweight
ORM, then the data-mapper ORMs---is supported at $\delta = 1.00$, $p = 0.012$. Because
each replicate is a freshly booted server process, these are independent replicates
rather than within-process repeats.

\subsection{Scaling with concurrency}
Figure~\ref{fig:scaling} sweeps the deep/nested fetch from 1 to 256 connections (a
separate single-run connection sweep). The
coarse three-band ordering (native and Prisma on top, the query builder and
lightweight ORM in the middle, the data-mapper ORMs below) holds at every operating
point at or above 8 connections; at a single connection the picture differs
(latency-bound, with several layers within noise of each other). Within the top
band, \texttt{pg} and Prisma alternate as the fastest layer: \texttt{pg} leads up
to 32 connections and Prisma from 64 onward, consistent with Prisma's higher parallelism
(and higher CPU use). Overall, the lightweight layers reach a markedly higher
throughput ceiling than the data-mapper ORMs across the load range, so the RQ1
finding is not an artifact of the single 50-connection operating point.

\subsection{Resource footprint}
Throughput is not the only cost (Table~\ref{tab:resources}). Prisma's near-native
throughput on the deep/nested fetch comes at a much higher CPU cost: measured at the
\emph{same} ten-connection pool as every other layer, its Node.js process draws
$471\%$ (about five cores) against the $\approx100\%$ (one core) of the others,
because Prisma runs a multi-threaded query engine (a Tokio runtime) in-process.
Because the pool is held constant, this is a property of the layer, not of a larger
connection pool. Prisma's throughput parity with native therefore trades CPU for
latency, which a throughput-only benchmark would not reveal
(Figure~\ref{fig:cpu_tradeoff}). Two caveats apply: this
is \emph{application-process} CPU only (database-side CPU lives in the
\texttt{postgres}/\texttt{mysqld} process and is excluded for every layer alike), and
the multiplier is pattern-dependent (Prisma's aggregation cell draws only $\approx
200\%$). Memory differences are smaller. The native driver, Knex, and Objection are
lightest (199--209~MB peak RSS), while the remaining ORMs use more (246--272~MB).
Resource cost, like throughput, is therefore pattern- and layer-specific, and
matters when sizing horizontally scaled deployments, where per-instance resource
use governs how many replicas a given load requires~\cite{vaquero2011scaling}.
