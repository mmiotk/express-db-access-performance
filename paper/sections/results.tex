\label{sec:results}

\texttt{paper/tables/} via \texttt{npm run sync:tables}. The numbers below come
from a run on the seeded dataset (2{,}000 authors, 100{,}000 posts, 1{,}000{,}000
comments; 50 concurrent connections; median of 3 measured runs after a warm-up
pass; benchmark-inserted rows reset before every cell). Absolute throughput is
hardware-specific; the comparison of interest is \emph{relative} across access
layers within an engine.

\paragraph{RQ1 (overhead vs native baseline).}
The native \texttt{pg} driver leads on every PostgreSQL pattern, but the cost of
abstraction is strongly \emph{pattern-dependent}. The lightweight query engine of
Prisma stays within a few percent of native on the deep/nested fetch
(3{,}538 vs 3{,}653~req/s) and even matches or exceeds native on aggregation, where
all layers issue the same raw correlated-subquery plan. In contrast, the data-mapper
ORMs pay a heavy price on the deep fetch, which materialises a nested object graph:
Sequelize, TypeORM, Objection and especially MikroORM (575~req/s) trail native by up
to $6.4\times$. Query builders (Knex) and the lightweight ORM (Drizzle) sit
consistently between these extremes.

\paragraph{RQ2 (engine dependence of the ranking).}
The \emph{relative} ordering of access layers is largely stable across PostgreSQL
and MySQL: native drivers and Prisma lead, data-mapper ORMs trail, on both engines.
The clearest engine-specific effect is on inserts, where MySQL throughput is
uniformly low (\char`~750--970~req/s across all layers, a $1.2\times$ spread)---the
InnoDB commit path dominates and masks access-layer differences---whereas on
PostgreSQL the write pattern spreads $4.6\times$ (native 7{,}120 vs MikroORM 1{,}544).
Thus the engine, not the access layer, is the primary determinant of write throughput.

\paragraph{RQ3 (which patterns matter most).}
Layer choice matters most for the deep/nested fetch (up to $6.4\times$ on PostgreSQL,
$4.1\times$ on MySQL) and least for aggregation once every layer uses the raw-SQL
escape hatch ($2.0\times$ / $1.4\times$). Point reads spread $3.5\times$ / $2.5\times$.
In short: the more of the result graph the access layer must assemble, the more the
choice of layer costs.
