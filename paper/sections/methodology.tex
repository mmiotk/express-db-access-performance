\section{Study Design}
\label{sec:methodology}

Our goal, in Goal--Question--Metric terms, is to \emph{analyze} the relational
database access layer of an Express.js service \emph{for the purpose of}
quantifying its performance cost \emph{with respect to} throughput and tail
latency \emph{from the point of view of} an application developer choosing a layer
\emph{in the context of} PostgreSQL and MySQL back ends~\cite{basili1988tame}. The
design follows established guidelines for controlled experimentation in software
engineering~\cite{wohlin2012experimentation,kitchenham2002guidelines} and crosses the
seven portable access layers with both engines and all five access patterns, plus
the two engine-specific native drivers and their two tuned counterparts,
\texttt{pg-tuned} and \texttt{mysql2-tuned} (an eleven-layer taxonomy: nine idiomatic
layers and two tuned lower-bound baselines); the complete rationale, including the
pitfalls each control guards against, is distributed with the replication package
(\texttt{METHODOLOGY.md}). The design targets two distinct estimands, kept separate
throughout: RQ1--RQ3 (Section~\ref{sec:results}) answer the
\emph{default-configuration} estimand, the cost a practitioner incurs from each layer
used idiomatically with its documented defaults, while the same-SQL control
introduced below answers a \emph{decomposition} estimand, what remains once every
layer executes identical SQL. Results are reported against whichever estimand they
speak to.

\subsection{Factors and treatments}
We vary three factors. The \emph{access layer} has eleven levels spanning the
taxonomy: the native drivers \texttt{pg} and \texttt{mysql2}, their tuned
counterparts \texttt{pg-tuned} (named prepared statements reused across calls) and
\texttt{mysql2-tuned} (the binary protocol with server-side prepared statements via
\texttt{execute()}), included as engine-specific lower-bound baselines rather than
idiomatic treatments; the query builder Knex; and the ORMs Drizzle (a lightweight,
query-builder-style ORM), Prisma, Sequelize, TypeORM, Objection, and MikroORM. The
nine idiomatic layers cover the taxonomy's three tiers but are a representative
cross-section, not an exhaustive catalogue; other libraries (for example Kysely,
Slonik, and pg-promise) are omitted. The \emph{engine} has two levels, PostgreSQL~18.4
and MySQL~9.7.1. The \emph{access pattern} has five levels, realized as HTTP
endpoints (Table~\ref{tab:patterns}). The four native and tuned baselines are
engine-specific (\texttt{pg}/\texttt{pg-tuned} run only on PostgreSQL,
\texttt{mysql2}/\texttt{mysql2-tuned} only on MySQL); the other seven layers run on
both engines, giving $4 + 7\times 2 = 18$ layer\,$\times$\,engine cells and
$18\times 5 = 90$ measured configurations.

\subsection{Objects: schema, workload, and data}
The workload is a minimal blog domain (\texttt{authors} $1\!-\!*$ \texttt{posts}
$1\!-\!*$ \texttt{comments} $*\!-\!1$ \texttt{authors}), which exercises a
one-to-many relationship and a two-hop join for the deep fetch while remaining
small enough to reason about. The five endpoints map onto the canonical access
patterns a CRUD service issues~\cite{cooper2010ycsb}: a primary-key point read; a newest-first
\emph{keyset} page (\texttt{WHERE id < cursor ORDER BY id DESC LIMIT}~$n$, using
the primary-key index rather than a large \texttt{OFFSET}); a deep/nested fetch
that returns a post together with its author and every comment, each with its own
comment-author; a per-author aggregation (post count, comment count, total views);
and a single-row insert. The database is loaded from a deterministic seed
(2{,}000 authors, 100{,}000 posts, 1{,}000{,}000 comments) by the native driver,
independently of any layer under test, so that every engine receives byte-identical
data. The working set fits in memory, so measurements reflect access-layer cost
rather than storage latency; once the working set is memory-resident, per-request
instruction and logging cost dominate execution rather than disk
I/O~\cite{harizopoulos2008oltp}. Every request draws its target identifier uniformly
at random from the full seeded range (posts $1$--$100{,}000$, authors
$1$--$2{,}000$), so the load spreads across the whole working set rather than
repeatedly hitting one hot record; because that set is memory-resident, the read
endpoints measure warm access-layer cost. A skewed, production-like key distribution
is a planned variant (Section~\ref{sec:threats}).

\subsection{The adapter contract and N+1 control}
Every access layer implements the same factory interface (five query methods plus
a teardown), so the Express server and the load runner are layer-agnostic. Each
adapter returns an \emph{identical result shape}, and each uses that layer's
\emph{recommended idiomatic} mechanism to avoid the N+1 query problem on the deep
fetch: a hand-written join for the native drivers, the Knex query builder, and the
query-builder-style Drizzle (the tuned baselines reuse the native drivers' join
unchanged), and relation eager loading (\texttt{include}, \texttt{withGraphFetched},
\texttt{populate}, or \texttt{relations}) for the data-mapper ORMs. The comparison is
therefore one of \emph{idiomatic usage} rather than of a single fixed query plan:
each layer runs the code its documentation recommends, so a measured difference
reflects the access layer together with the SQL it generates, its round-trip count,
and how it materializes results. One asymmetry is inherent to any driver-versus-ORM
comparison: the native side's idiom is hand-authored SQL, whereas the ORM side's
idiom is the library's default; the same-SQL control below quantifies how much of
the difference that choice accounts for. The number of round trips is itself part of
what varies: the native drivers, Knex, and Drizzle issue a two-query deep fetch,
whereas each ORM's eager-loading strategy selects its own query count and join
shape. Server-side statement logging captures the exact SQL each layer emits, its
round-trip count, and \texttt{EXPLAIN} plans for the native statements; these are
released with the replication package, and the online supplement tabulates the
per-endpoint counts. They differ by design: the deep fetch takes one query in
Sequelize and MikroORM, two in the native drivers, Knex, Drizzle, and TypeORM, and
four in Prisma and Objection. To separate a layer's execution machinery from its
strategy choice, every adapter also implements a \emph{same-SQL control}: identical
two-statement deep-fetch SQL (and, per engine, identical \texttt{EXPLAIN} plans for
those statements) with identical row mapping, executed through the layer's raw-SQL
facility, mirroring the aggregation control (results in Section~\ref{sec:results}).
Server-side statement capture, synchronized to a request marker so connection
handshakes are excluded, confirms the two control statements are byte-identical
across every layer on both engines; what differs is the wire protocol each driver
negotiates for them: on PostgreSQL every layer uses the extended protocol with
server-side prepared statements except MikroORM, which falls back to the simple
protocol with client-side literal inlining, and on MySQL every layer uses the text
protocol except \texttt{mysql2-tuned} and Prisma, which use the binary protocol with
server-side prepared statements. To guarantee that no adapter gains an unfair
advantage by silently returning less, or by issuing N+1 queries, an automated
cross-check (\texttt{bench/verify.mjs}) funnels every adapter's rows through shared
canonical constructors and asserts full JSON byte equality against the
native-driver baseline, on both engines, across 12 probes per adapter spanning the
four non-mutating patterns and the same-SQL control, before any timing is collected.
This check caught two genuine defects during development (a fan-out aggregation bug
and a driver result-shape mismatch; see Section~\ref{sec:discussion}).


\subsection{Controls}
Four controls isolate the access layer as the only variable. \emph{(i)~Pool size}
is fixed at ten connections for every adapter, so the comparison measures
access-layer overhead rather than pool tuning, itself a known web-application
performance factor and a recurring concern in database-backed
services~\cite{gupta2017pooling,laigner2021microservices}; for Prisma, whose default pool is
otherwise derived from the core count, this is pinned explicitly through the
\texttt{connection\_limit} URL parameter. Because the load generator opens 50
concurrent connections against this ten-connection pool, roughly forty in-flight
requests are always waiting in each library's connection-acquisition queue; that
queueing path is deliberately part of what we measure, since it is exactly what a
loaded production service exercises, and it is held structurally identical (same
pool size, same demand) across layers. A 200~ms connection-pool sampler corroborated
this directly on the deep fetch: the native PostgreSQL cell ran with its
ten-connection pool fully occupied (mean utilization $\approx 10$ of 10) and up to
39--40 requests pending at once, direct evidence of the connection-acquisition queue
rather than an assumption about it. \emph{(ii)~Identical query
semantics}: each endpoint issues the same logical query across layers; the
aggregation, in particular, uses a single correlated-subquery formulation
everywhere (the documented raw-SQL path), so that endpoint measures layer
overhead on an identical plan rather than differences in generated SQL.
\emph{(iii)~Write isolation}: because the insert endpoint mutates the dataset, every
cell resets it before warm-up and before every measured run, not merely once per
cell; the write endpoint is additionally measured in its own dedicated server boot,
taken after the database's physical state is rebuilt rather than merely emptied by
row deletion (PostgreSQL: drop and recreate the database from a seeded template;
MySQL: delete the benchmark rows, rebuild the table with \texttt{OPTIMIZE TABLE},
and re-pin \texttt{AUTO\_INCREMENT}), so sequences, physical layout, and purge state
cannot drift across replicates or leak between adapters. \emph{(iv)~Observer
separation}: the HTTP load generator runs in a process separate from the server.

\subsection{Measurement procedure}
Load was generated with \texttt{autocannon}~\cite{autocannon}, an HTTP/1.1 load
tester that issues requests continuously at a fixed concurrency and records a
latency histogram; it is a closed-loop (constant-concurrency)
generator~\cite{schroeder2006open}. We measured each of the 90 configurations as 25
\emph{independent replicates}. For each replicate we booted a fresh server process,
opened 50 concurrent connections, ran a fifteen-second warm-up pass whose
measurements were discarded, and then took one twelve-second measured run. The
warm-up length follows a separate cold-boot measurement that logged per-second
throughput from server start for a fast, a middling, and the slowest PostgreSQL
layer: every layer reached and sustained at least 95\% of its steady-state
throughput within 12~s of a cold boot (Prisma by 5~s, the native driver by 9~s,
MikroORM, the slowest, by 12~s), so the fifteen-second warm-up begins each
measurement past the slowest observed ramp (absorbing JIT compilation,
connection-pool fill, and query-plan caching, since managed runtimes do not
always reach a stable steady state promptly~\cite{barrett2017warmup}); the
few-percent residual fluctuation around the plateau is bounded separately by the
sustained-load check in Section~\ref{sec:threats}. Booting a fresh process per replicate makes the
replicates statistically independent rather than repeated measures within one
process lifetime, and cell order was reshuffled independently within each replicate;
a forward-versus-reversed-order check on the write endpoint (5 replicates per
direction) found no detectable history effect (reversed-to-forward throughput ratio
across all 18 cells: 0.970--1.021, median 1.004), supporting the shuffled design.
Within a replicate, every layer and engine is driven by the identical request-id
sequence, generated by a PRNG seeded from the (endpoint, replicate) pair alone; this
paired-stream design removes between-layer sampling noise as a confound, and the
first ids of every stream are archived in the replication package. We report the
median of the 25 replicate values; 449 of the 450 dedicated write-endpoint boots in
the primary campaign completed normally, the exception being knex/PostgreSQL, whose
replicate-16 boot hit a health-check timeout immediately after the physical database
rebuild, leaving that cell with 24 of 25 replicates (the runner records and skips
such failures rather than retrying silently). A port allocator that skips the ports
the databases themselves listen on prevents the collision that had cost an earlier
replicate its server process. Each replicate yields throughput (requests per second)
and latency percentiles p50, p90, p97.5, and p99. During the measured runs we
sampled server CPU utilization and peak resident memory from \texttt{/proc} over the
full process tree (parent plus descendants, so an in-process engine or a spawned
helper cannot escape the measurement), with database and load-generator CPU sampled
separately for comparison; a \texttt{perf\_hooks} GC observer (event count, total
pause time) and the 200~ms connection-pool sampler mentioned above ran inside the
server and were polled through a \texttt{/stats} endpoint after each run. Both
engines ran with the working set resident in memory. Inserts were measured under
each engine's \emph{default} durability configuration (PostgreSQL
\texttt{fsync=on}, \texttt{synchronous\_commit=on}, \texttt{full\_page\_writes=on};
MySQL \texttt{innodb\_flush\_log\_at\_trx\_commit=1}, \texttt{sync\_binlog=1},
\texttt{log\_bin=ON}), the regime a deployment runs unless an operator deliberately
relaxes it; a labelled secondary run instead relaxed every one of those settings on
both engines (10 replicates, write endpoint only) to isolate the durability
mechanism, compared with the default regime in Section~\ref{sec:discussion}
(Table~\ref{tab:durability}). A no-database \texttt{/baseline} endpoint returning a
fixed, seed-realistic thread-shaped payload measured the shared
Express-plus-serialization floor: 10{,}909 (PostgreSQL server) and 10{,}921 (MySQL
server) req/s on this host, roughly $3\times$ the fastest deep-fetch cell, so the
framework floor leaves the between-layer spreads visible rather than compressing
them. The hardware and software environment was captured automatically into the
replication package. To characterize behavior under load, we additionally swept the
deep/nested fetch (the most layer-sensitive pattern) across connection counts from 1
to 256 for every layer and engine, since throughput and contention scale
non-linearly with concurrency~\cite{yu2014concurrency}; a separate equal-CPU-budget
control, confining the server process to 1, 2, or 4 cores, attributes part of that
scaling directly to CPU (Section~\ref{sec:results}). A native, \texttt{undici}-based
constant-arrival generator drove the deep fetch under a fixed offered rate
(250--4{,}000~req/s) rather than fixed concurrency, measuring every latency from the
request's intended send time and clipping timed-out requests into the distribution
at their cutoff; this coordinated-omission-corrected
design~\cite{tene2015latency} validates the closed-loop tail results independently
(Section~\ref{sec:results}).

\subsection{Analysis}
Because absolute throughput is hardware-specific, we analyze \emph{relative}
differences across layers within an engine. We quantify a pattern's sensitivity to
the access layer by its \emph{spread}: the ratio of the idiomatic native driver's
(\texttt{pg} or \texttt{mysql2}; the tuned baselines are a separate lower-bound
reference, not the spread's anchor) median throughput to that of the slowest layer
on that pattern (native-relative, so it is comparable across patterns even where one
ORM exceeds the native driver). We report medians over the 25 independent
replicates, the coefficient of variation (CV) across them as a stability signal,
following the practice of reporting central tendency and variability rather than a
single number~\cite{georges2007rigorous} and of using enough replicates to bound
variance~\cite{laaber2019microbenchmarking} (Section~\ref{sec:results}), and a
bootstrap 95\% confidence interval for the median. To test whether adjacently
ranked layers are distinguishable rather than an artifact of measurement noise, we
compare each adjacent pair on the deep/nested fetch with the Mann--Whitney $U$ test
(two-sided) and report Cliff's~$\delta$ as a nonparametric effect size, the latter
being robust to the small number of replicates and to non-normal throughput
distributions; reporting a nonparametric test together with a standardized effect
size follows recommended practice for randomized performance experiments in software
engineering~\cite{arcuri2014hitchhiker,cliff1993dominance}. We do not
apply significance testing to widely separated layers, where the effect is obvious;
the tests target the close pairs where a ranking could plausibly be noise.

\subsection{Experimental setup}
All measurements were taken on 2026-07-12 and 2026-07-13 on a single host: the
primary matrix ran overnight across both dates, and the order-invariance,
durability, same-SQL, open-loop, fan-out, equal-CPU, and concurrency-sweep runs
followed on 2026-07-13. All ran on the same host
(Intel Xeon Gold~6140, 16~cores, 126~GB RAM; Linux~6.17; Node.js~24.16.0) with the
database engines running locally. The pinned versions (Table~\ref{tab:versions})
were the current releases at that date and were frozen for the study, so later
point releases (for example Node.js~24.17.0) are deliberately excluded. We pin
Express~4, still the dominant deployed line at measurement time, rather than
Express~5; the framework version is shared by every cell, so it shifts the common
floor rather than the between-layer comparison, and re-running the harness on
Express~5 is a one-line change. Because access-layer performance is version-sensitive, we
pin every dependency (Table~\ref{tab:versions}); these are the versions the reported
numbers characterize, and re-running the harness on newer releases is a one-command
operation.

\subsection{Replication package}
The harness (the Express application, the adapter contract, all nine idiomatic
adapters and the two tuned baselines, the deterministic seed, the
\texttt{autocannon} matrix runner, the correctness cross-check, and the scripts that
regenerate every table in Section~\ref{sec:results}) is released so the full matrix
can be re-run with a single command against a containerized or local PostgreSQL and
MySQL.
