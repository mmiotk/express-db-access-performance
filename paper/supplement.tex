\noindent This supplement carries measurement detail referenced from the main
text. All tables are generated by the replication package
(\url{https://github.com/mmiotk/express-db-access-performance}; archived at
\url{https://doi.org/10.5281/zenodo.21339518}) from the same raw data as the
main-text tables.

\section{Dispersion on MySQL}
\label{sup:cv-mysql}
The main text reports the coefficient-of-variation table for PostgreSQL and
the per-engine maxima. Table~\ref{tab:cv_mysql} gives the full per-layer,
per-pattern CV on MySQL across the 25 independent replicates (maximum 6.6\%,
\texttt{knex}/insert).

\input{\tabledir cv_mysql}

\section{Idiomatic round-trip counts}
\label{sup:query-counts}
Table~\ref{tab:query_counts} reports, for each layer, the number of SQL
statements the idiomatic deep fetch issues per request, captured from
server-side statement logs. The counts motivate the main text's observation
that round-trip count alone does not determine throughput: the two layers
issuing four statements (Prisma, Objection) sit at opposite ends of the
throughput ladder.

\input{\tabledir query_counts}

\section{Write throughput under default versus relaxed durability}
\label{sup:durability}
Table~\ref{tab:durability} gives the per-layer insert throughput under each
engine's default durability configuration (the paper's primary write regime)
and under the symmetric relaxed regime (secondary, 10 replicates), with the
relaxed$\div$default ratio quoted in the main text.

\input{\tabledir durability}

\section{Equal-CPU-budget control}
\label{sup:equalcpu}
Table~\ref{tab:equalcpu} gives the deep-fetch throughput with the server
confined to 1, 2, or 4 cores (database and load generator pinned to disjoint
cores); all nine values are quoted in the main text.

\input{\tabledir equalcpu}

\section{Application-tier versus end-to-end CPU efficiency}
\label{sup:cpu}
Table~\ref{tab:cpu_efficiency} reports, per layer on the deep/nested fetch,
the median application-process-tree CPU and database-server CPU, the combined
core-count, and throughput per application-CPU-second and per combined
(application$+$database) CPU-second. The application-only figure understates the
compute of layers that delegate more work to the database: \texttt{pg} leads
application-tier efficiency by roughly $5\times$ over Prisma (3{,}741 vs.\
740~req per app-CPU-s) but only $1.4\times$ end-to-end (813 vs.\ 596~req per
combined-CPU-s), because it pushes about 3.6 database cores of work that the
application-only metric ignores.

\input{\tabledir cpu_efficiency}

\section{Coordinated-omission-corrected open-loop sweep}
\label{sup:openloop}
Table~\ref{tab:openloop} gives the full constant-arrival sweep (PostgreSQL,
five layers, offered rates 250--4{,}000 requests/s) behind the open-loop
paragraph of the main text: achieved throughput, corrected latency
percentiles measured from each request's \emph{intended} send time, and
timeout counts (10~s cap, clipped into the distribution).

\input{\tabledir openloop}
